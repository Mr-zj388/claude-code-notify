# Codebase Navigator

生成和维护 `CLAUDE.md` + 镜像 `docs/ai-nav/` 目录。

---

## 语言规则（最高优先级）

**所有生成的文档内容必须使用中文编写。** 包括：
- `CLAUDE.md` 中的所有标题、描述、规则说明
- `docs/ai-nav/` 中所有 `.nav.md` 和 `_index.nav.md` 的标题、描述、表格内容
- 禁止事项的描述文字

**保持英文的内容：**
- 代码标识符（类名、方法名、变量名）
- 文件路径和目录名
- 包名和技术术语缩写（如 URP、RTS、A*）
- 代码块内容

---

## 触发规则（关键）

本 skill 必须在以下场景自动触发：

### 自动触发 1：提交前同步
**时机**：用户要求提交代码（git commit、/commit 等）
**在**创建提交之前，执行**增量模式**：
1. 检测哪些 .cs 文件被添加/修改/删除（通过 `git diff --cached --name-only`）
2. 仅为变更的文件重新生成 `.nav.md`
3. 为有文件增删的目录更新 `_index.nav.md`
4. 暂存更新的文档：`git add docs/ai-nav/`
5. 然后继续提交

### 自动触发 2：CLAUDE.md 修改
**时机**：任何操作会修改 `CLAUDE.md`（用户要求更新规则、添加禁止事项、修改架构等）
**动作**：修改 CLAUDE.md 后，检查 `docs/ai-nav/_index.nav.md` 是否需要对应更新（如新增模块、目录规范变更）。保持 CLAUDE.md 在 280 行以内 —— 如内容过于详细，移至对应的 `_index.nav.md`。

### 自动触发 3：文件结构变更
**时机**：创建新的 .cs 文件、文件重命名或文件删除时。
**动作**：创建/更新/删除对应的 `.nav.md` 并更新父级 `_index.nav.md`。

### 手动触发
**时机**：用户明确调用 `/codebase-navigator` 或说"更新文档"、"同步导航文档"、"重新生成 CLAUDE.md"
**动作**：根据用户请求执行完整模式或增量模式。

---

## 执行模式

### 完整模式（首次或明确要求完整重新生成）

1. **扫描项目配置** — 读取项目文件，检测技术栈、命令和结构。
2. **映射源码树** — 列出目录和文件，跳过"跳过规则"中的项目。
3. **读取关键文件** — 浏览入口点、核心模块以理解架构。
4. **发现禁止事项** — 从一致的代码模式中推断规则。
5. **创建 `docs/ai-nav/` 目录** — `mkdir -p docs/ai-nav/{mirror paths}`。
6. **生成 `CLAUDE.md`** — 280 行以内。结构化路由器，包含模块路由表。**必须使用中文。**
7. **生成导航文档** — 每个文件的 `.nav.md` + 每个目录的 `_index.nav.md`。**必须使用中文。**
8. **报告** — 文件数量，提醒审查，询问 .gitignore。

### 增量模式（提交时或文件变更后）

1. **检测变更** — `git diff --cached --name-only` 或 `git diff --name-only`。
2. **过滤源文件** — 仅处理 `.cs`（或主要语言）文件，跳过生成/配置文件。
3. **对每个变更的文件**：
   - 如果**新增**：读取文件，创建 `.nav.md`，更新父级 `_index.nav.md`。
   - 如果**修改**：重新读取文件，重新生成 `.nav.md`（尽可能保留手写的"修改指南"部分）。
   - 如果**删除**：删除其 `.nav.md`，更新父级 `_index.nav.md`。
4. **暂存文档变更** — `git add docs/ai-nav/`（仅在提交前）。
5. **简要报告** — "已为 M 个变更文件更新了 N 个导航文档。"

## 6 大核心原则

### 1. 路由器不是百科

CLAUDE.md 是**指针**，不是知识库。它告诉 Claude **去哪里看**，而不是把所有内容都写出来。

- 错误示范：在 CLAUDE.md 中粘贴 50 行 API schema 文档
- 正确示范：`API schema 定义在 src/schemas/ 中。修改任何接口前先阅读对应的 schema 文件。`

导航文档 (`docs/ai-nav/`) 的存在就是为了让 CLAUDE.md 保持精简。CLAUDE.md 负责路由，导航文档负责详情。

### 2. 禁止 > 建议

禁止事项是可执行的。建议在压力下会被忽略。始终优先使用"禁止做 X"而非"尽量做 Y"。

将禁止事项格式化为独立章节，规则清晰可扫描。

### 3. 只索引你的代码

记录**你项目**的代码，而非库的用法。Claude 已经了解 React、Express、Django、Unity ECS。

导航文档中的外部依赖：仅列出包名，不做解释。

### 4. 可验证

如果一条规则无法检查合规性，它就是噪音。每条指令都应有明确的通过/失败条件。

测试：能否写一个 lint 规则或 grep 命令来检查？如果能，就是可验证的。如果不能，重写它。

### 5. 分层

```
~/.claude/CLAUDE.md                    # 全局：个人默认设置
project/CLAUDE.md                      # 项目级：团队共享规则（<280行）
project/src/auth/CLAUDE.md             # 目录级：模块特定覆盖
project/.claude/CLAUDE.local.md        # 个人本地：gitignore 排除
docs/ai-nav/                           # 参考映射：按需读取，非会话启动时加载
```

仅当模块的规则与项目级规则冲突或扩展时，才生成目录级 CLAUDE.md。

### 6. hooks 兜底

CLAUDE.md 中的规则是尽力而为的。对于每次都**必须**执行的事项，推荐使用 hooks。

#### CLAUDE.md 生成规则

1. **280 行以内。** 保持结构化和可扫描 —— 包含模块路由表、ECS 约定和关键注意事项，但将逐文件细节移至 `docs/ai-nav/`。
2. **禁止事项章节是必须的。** 从代码模式中找到至少 3 条真实的禁止规则。
3. **每条规则都必须可验证。** 能否用 grep/lint/test 检查违规？如果不能，重写或删除。
4. **命令必须真实。** 只列出项目配置中实际存在的命令。永远不要编造命令。
5. **路由，不解释。** 指向文件和目录。不解释其中的内容。
6. **不写外部库文档。** 告诉 Claude 你的代码在哪里，而非库怎么用。
7. **删除空章节。** 永远不出现 `_TODO_` 占位符。
8. **全部使用中文。** 标题、描述、规则说明一律中文，仅代码标识符和路径保持英文。

### 如何发现禁止事项

- 如果每个文件都用 logger 而非 console.log → `禁止使用 console.log`
- 如果有错误工具类且无原始 throw → `禁止抛出原始 Error 对象`
- 如果所有组件都遵循命名模式 → `禁止命名组件时不加模块前缀`
- 如果有常量文件 → `禁止硬编码魔法数字`

查看 CI/CD 配置、linter 配置和现有代码审查注释以获取更多线索。

## 跳过规则

**永远不为以下内容生成导航文档：**
- `node_modules/`、`.git/`、`__pycache__/`、`dist/`、`build/`、`.next/`、`target/`、`vendor/`、`.venv/`
- `Library/`、`Temp/`、`Logs/`、`obj/`（Unity/C# 构建产物）
- 锁文件、二进制文件、图片、字体、编译输出、source maps
- `TextMesh Pro/`（Unity 包，非项目代码）
- 不言自明的配置文件
- `.gitignore` 中的任何内容

**始终为以下内容生成导航文档：**
- 主要语言的所有源代码（Unity 项目为 `.cs`）
- 包含业务逻辑的配置文件（asmdef、ScriptableObject schema）
- Editor 脚本和工具

## 大型项目（100+ 文件）

1. 询问用户哪些模块最重要
2. 先为核心模块生成完整细节
3. 为外围模块生成轻量摘要（跳过"修改指南"部分）
4. 逐目录工作，报告进度

---

## 更新现有文档

如果 `docs/ai-nav/` 已存在：
1. 仅为变更的文件重新生成导航文档
2. 如有文件增删则更新 `_index.nav.md`
3. 除非明确要求，否则不重新生成 `CLAUDE.md` —— 它可能包含手动编辑的内容
4. 保留现有导航文档中手写的"修改指南"部分
